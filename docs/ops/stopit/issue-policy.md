# Stopit Issue Policy

## 기본 원칙

- GitHub Issue는 한국어로 작성한다.
- GitHub Issue는 실행 가능한 작업 단위여야 한다.
- 분석 메모, 아이디어, 가설은 바로 Issue로 만들지 않는다. 필요하면 Discord 아이데이션으로 보낸다.
- 기존 열린 이슈와 중복되면 새로 만들지 않는다.
- 이슈 생성 전 항상 열린 이슈를 조회한다.
- 외부 side effect는 메인 오케스트레이터만 수행한다. subagent는 분석만 한다.

## 생성 제한

일일 자동 생성 제한:
- 지표 기반 이슈: 최대 3개
- 저장소 건강도 이슈: 최대 5개
- 아이디어를 GitHub Issue로 전환: 최대 2개

제한은 백로그 소음을 줄이기 위한 상한이다. 근거가 부족하면 0개가 맞다.

## 라벨 정책

가능한 기존 라벨을 우선 사용한다.

주요 라벨:
- `bug`
- `refactor`
- `tech-debt`
- `qa`
- `performance`
- `maintenance`
- `documentation`
- `automation`
- `analytics`
- `product-metrics`
- `growth`
- `monetization`
- `priority:p0`
- `priority:p1`
- `priority:p2`

우선순위 기준:
- `priority:p0`: 사용자 안전/잠금/긴급해제/배포/데이터 손실/심각한 크래시 가능성
- `priority:p1`: 핵심 활성화, 반복 사용, 계측 신뢰, CI/release blocker, 중대한 품질 리스크
- `priority:p2`: 개선 여지가 크지만 즉시 장애는 아닌 작업, 문서/리팩터링/실험 준비

## 제목 형식

지표/제품:
- `[지표] 구체적인 작업명`
- `[성장] 구체적인 작업명`
- `[수익화] 구체적인 작업명`
- `[리뷰] 구체적인 작업명`

저장소 건강도:
- `[버그] 구체적인 작업명`
- `[QA] 구체적인 작업명`
- `[기술부채] 구체적인 작업명`
- `[리팩터링] 구체적인 작업명`
- `[유지보수] 구체적인 작업명`
- `[문서] 구체적인 작업명`

좋은 제목:
- `[지표] 첫 잠금 활성화 퍼널의 버전별 계측 신뢰도 검증`
- `[QA] 긴급해제 카운트다운 상태 전환 회귀 테스트 보강`

나쁜 제목:
- `지표 개선`
- `TODO 정리`
- `앱 좋게 만들기`

## 지표 기반 Issue 본문

```md
## 지표/근거
- 분석 기간:
- 비교 기간:
- 핵심 수치:
- 확인 소스:

## 문제

## 제안 작업
- 작업 1
- 작업 2
- 작업 3

## 완료 기준
- [ ] 측정 가능한 완료 조건
- [ ] 구현/문서/테스트 조건
- [ ] 14일 또는 30일 후 전후 비교 조건
```

- 관찰 내용:

- 작업 3

## 아이디어/실험 Issue 본문

```md
## 문제/기회

## 제안 기능 또는 실험

## 기대 효과

## 리스크/가드레일

## 완료 기준
- [ ] 실행 가능한 산출물
- [ ] 측정 지표 또는 검증 방법
- [ ] 출시 후 확인 기간
```

## 중복 판단 기준

중복으로 본다:
- 같은 사용자 문제를 같은 방식으로 해결한다.
- 같은 파일/기능 영역에서 같은 완료 기준을 요구한다.
- 기존 이슈의 하위 작업으로 충분히 흡수 가능하다.

중복이 아닐 수 있다:
- 같은 영역이지만 다른 사용자 문제다.
- 기존 이슈는 분석/계측이고 새 후보는 구현/QA다.
- 기존 이슈가 너무 커서 명확한 작은 slice로 분리해야 한다.

중복 의심 시:
- 새 이슈를 만들지 말고 최종 보고에 “중복/기존 이슈 참조”로 남긴다.
- 필요하면 기존 이슈에 comment를 남기는 것은 대표님이 원할 때만 한다.

## 생성 후 검증

이슈 생성 후 반드시 확인한다.

```bash
gh issue list --state open --limit 20 --json number,title,labels,url
```

보고에는 생성한 URL과 라벨을 포함한다.
